# Architecture — Kali BMH Medallion Pipeline

## Overview

This project implements the **Medallion Architecture** (Bronze → Silver → Gold)
for a B2B industrial equipment sales dataset themed around Kali BMH Systems,
a bulk material handling manufacturer based in Tamil Nadu, India.

The pipeline runs on two paths that share identical transformation logic:

- **Cloud path** — Databricks notebooks + Unity Catalog + Serverless compute
- **Local path** — `run_local.py` + local Delta Lake under `./lakehouse/`

---

## Layer design

### Bronze — raw ingestion

The Bronze layer is append-only and schema-on-read. Two sources land separately:

- **CSV** — 3,000 rows generated by `src/data_generator.py` with a realistic
  growth trend and seasonal pattern. Uploaded to a Databricks Volume
  (`/Volumes/kali_demo/bronze/raw_files/`) for the cloud path.
- **PostgreSQL** — 500 rows seeded into a Dockerized Postgres 16 container.
  Exposed to Databricks via an ngrok TCP tunnel and read via JDBC.
  This simulates a hybrid cloud/on-premise ingestion pattern.

Both sources land as separate Delta tables — `sales_orders_csv` and
`sales_orders_postgres` — with `ingested_at` and `source` audit columns added.
No transforms are applied at this stage.

### Silver — cleaned and trusted

The Silver layer applies four transformations in sequence:

1. **Schema alignment** — drops `created_at` from the Postgres source
   (a DB-internal column absent from the CSV source) before union
2. **Union** — `unionByName` combines both Bronze sources into 800 rows
3. **Deduplication** — a Window function partitioned by `order_id` and
   ordered by source priority (Postgres = 1, CSV = 2) keeps one row per
   order. Postgres is the master source — when the same `order_id` exists
   in both, Postgres wins. Result: 500 unique rows
4. **Cleaning** — `trim()`, `initcap()`, `upper()` standardise string columns;
   empty strings are converted to `NULL`

A `MERGE INTO` (upsert) writes to the Silver table — idempotent by design.
Re-running the pipeline any number of times produces the same result.

### Gold — business aggregations

Four aggregation tables serve specific business questions:

| Table | Business question |
|---|---|
| `revenue_by_region` | Which Indian states drive the most revenue? |
| `revenue_by_product` | Which Kali BMH products are top sellers? |
| `order_status_breakdown` | How much revenue is at risk (PENDING/CONFIRMED)? |
| `monthly_sales_trend` | How has revenue trended month over month? |

Delta table constraints on `revenue_by_region` enforce data quality at the
storage layer — negative revenue or zero-order rows are rejected on write.

### Forecast — ML capstone

A Prophet time-series model is trained on `monthly_sales_trend` to forecast
the next 6 months of revenue. Key design decisions:

- **Additive seasonality** — more stable than multiplicative for short series
- **Quarterly seasonality** — 91.25-day period captures B2B buying cycles
  without requiring 2+ years of data for yearly seasonality
- **MLflow tracking** — params, MAE, MAPE and the forecast chart are logged
  to `./mlruns/` locally or Databricks managed MLflow in the cloud
- **Gold Delta table** — `revenue_forecast` stores predictions as a
  first-class Gold asset with `yhat`, `yhat_lower`, `yhat_upper` columns

## Deduplication strategy

The Window function approach was chosen over simpler alternatives:

- `dropDuplicates()` — non-deterministic when duplicates exist across sources;
  no way to specify which source wins
- `MERGE INTO` alone — handles upserts but not cross-source conflict resolution
- **Window + row_number** — deterministic, source-priority-aware, and testable

The test suite in `tests/test_silver.py` verifies this with a hand-built
fixture asserting Postgres wins on `order_id=1` conflict.

---

## Shared package design

All transformation logic lives in `src/medallion/` — an importable Python
package used by both the Databricks notebooks and `run_local.py`. This ensures:

- **No drift** — cloud and local paths run identical code
- **Testability** — pure functions taking and returning DataFrames
- **Single source of truth** — change `silver.py` once, both paths update
